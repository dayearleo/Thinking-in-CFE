# CFE 算子超越点 · 3 维评估

> **触发**:user 追问 "我们超越未来 FT QC 的地方在哪里? · 我们实现了 quantum-only 才能存在的算子吗?"
> **状态**:超越点定位 v1 · 是 dev-notes/004 算子定义的对比分析篇
> **真理源**:本文件 = $\Phi^{CF}_f$ 跟其他 paradigm 比较的单点真理源

## 0 · 起源

定义算子 (`dev-notes/004`) 后 · 立即面临一个 strategic question:**它跟未来 FT QC 是什么关系? 跟现有 Grover / quantum walk 是什么关系?**

简单答 "我们更好" 是错的 · 简单答 "我们是 FT QC 子集" 也是错的。本文件提出 3 维评估框架 · 在不同维度上分别给精确定位。

## 1 · 三维评估框架

把"超越"拆成 3 个独立维度 · 否则全是混乱比较:

| 维度 | 评什么 |
|---|---|
| **D1 · capability** | 算子能算什么 · 给什么保证 |
| **D2 · cost** | 需要多少硬件 / 量子资源 |
| **D3 · interface domain** | 什么样的 oracle 能被这个算子作用 |

把所有"超越"的讨论都先归到这三维之一 · 再做对比。

## 2 · D1 capability 对比

### 2.1 · 算子能力 (能算什么 Boolean 函数)

| 算子族 | 可算的函数 $f$ |
|---|---|
| 经典 OR/AND/XOR 等 | 任意 Boolean |
| Grover oracle queries | 任意 oracle 包装的 Boolean |
| Standard quantum query (Q(f)) | 任意 Boolean |
| **$\Phi^{CF}_f$** | 任意 Boolean |
| FT QC (gate model) | 任意 Boolean + 任意 unitary |

**结论 (D1 capability)**:$\Phi^{CF}_f$ 跟经典/标准量子在 Boolean 函数层面**等价** · 比 FT QC 子集。**我们不在 D1 上超越 FT QC**。

### 2.2 · D1 隐藏维度 · "查询的可观察性"

但有一个被忽视的 D1 子维度:**查询行为本身是否对持有 oracle 的对手可见**?

| 算子族 | 持 oracle 对手能否检测查询发生? |
|---|---|
| 经典 | ✅ 必然可见 (查询就是经典通信) |
| Standard quantum query | ✅ 可见 · oracle 跟主系统纠缠 · 纠缠可测 |
| **$\Phi^{CF}_f$** $(\delta \to 0)$ | ❌ **不可见** · 没有纠缠产生 |
| FT QC implementing same | ✅ 仍可见 · 因为 unitary oracle calls 必然纠缠 |

**关键发现**:在这个 D1 子维度上 · **$\Phi^{CF}_f$ 超越 FT QC**。FT QC 即使无限大也无法做"对手不可检测的查询" · 因为 unitary 调用本质上产生纠缠。

## 3 · D2 cost 对比

### 3.1 · 实现单个 $\Phi^{CF}_f$ 调用的硬件成本

| 实现方式 | 物理 qubit/资源 | 错误率要求 | 时钟 | 平台规模 |
|---|---|---|---|---|
| FT QC 实现 $\Phi^{CF}_f$ | ~$10^3 N$ 物理 qubit (含 surface code overhead) | 物理 $10^{-3}$ → 逻辑 $10^{-15}$ (需 ~$10^5$ 表面码循环) | 逻辑门 ~μs | 全球各 ~$10^{10}$ 投资 |
| **专用 photonic IFM** (我们的) | N 路 photonic mode | 5 dB loss + visibility 95% 足够 | 飞行时间 ~ns | 单芯片 $50k-150k |

**比例**:大约 6 个数量级的成本差。

### 3.2 · 这跟 FFT 历史很像

| 时代 | 通用方案 | 专用方案 | 差异 |
|---|---|---|---|
| 1960s-70s · FFT 兴起 | 数字 CPU 跑 Cooley-Tukey | 光学 4f 透镜系统 | 光学瞬时 · 但只能做 FFT |
| 1990s · DSP 出现 | 通用 CPU 浮点 | 专用 DSP 芯片 | DSP 拿下 modem / 音频 |
| 2000s · GPU | 多核 CPU | GPU | GPU 拿下 3D / 后来 AI |
| **当前 · counterfactual** | FT QC (未来) | **专用 photonic IFM (我们的)** | 提前 unlock · 抢占 niche |

每一代 "专用先到 · 通用后追 · 专用永远占某 niche" 是稳定模式。我们处在 counterfactual 这个新原语的 "专用先到" 期。

## 4 · D3 interface domain 对比 (核心超越点)

这一维度是真正的 categorical 差异。

### 4.1 · 谁能当 oracle?

| 算子族 | oracle 必须是什么 |
|---|---|
| FT QC | 自己构造的 quantum circuit (内部) |
| Standard quantum query (theory) | 黑盒 quantum oracle (内部抽象) |
| **$\Phi^{CF}_f$ (我们)** | **任何能跟 photon 物理耦合的真实物体** |

### 4.2 · D3 上 $\Phi^{CF}_f$ 独占覆盖的 oracle 类型

下面这些 oracle · FT QC 永远到不了 (因为它需要把 oracle 放进自己的 quantum register):

- 一根**真实**爆炸物的引信状态
- 一颗**真实**稀土晶体里某核素的存在
- 一束**真实**对手雷达光是否在工作
- 一个**真实**生物细胞当前是否活着
- 一片**真实**晶圆某 die 是否有缺陷
- 一束**真实**远程激光信号的存在性
- 任何**真实物理对象**的某个 binary 属性

FT QC 要查询这些 · 必须先**测量它们** → 但测量本身 destructive · 反事实性丢光。

而 $\Phi^{CF}_f$ 天然能接 —— photon path 直接物理穿过 / 反射 / 吸收 物理样本 · 不需要数字化先。

### 4.3 · D3 是 categorical 差异

D1/D2 都是程度差异 (FT QC 可以"更好" / 我们可以"更便宜")。
**D3 是范畴差异** —— FT QC 跟物理世界的 interface 必须经过测量这道关卡 · 而测量破坏反事实性。

类比:

- **CPU vs Sensor** · CPU 再强大也不能直接感受温度 · 必须经过 sensor 接口 · sensor 永远不可替代
- **Memory vs Cache** · Cache 不是 "更小的 memory" · 是有局部性约束的特殊 memory · 范畴不同
- **Compiler vs Linker** · Linker 不是 "另一种 compiler" · 它处理 compiler 处理不了的跨文件符号解析

我们在 D3 上对 FT QC 的关系类似 sensor 对 CPU · **不互相替代 · 互相需要**。

## 5 · 三个超越点详论

综合 D1 子维度 + D3 的发现 · 我们对 FT QC 有 3 个真正不可绕过的优势点:

### 超越点 1 · D3 · 外部物理 oracle 的反事实查询

- **谁有**:只有我们 (集成 photonic IFM)
- **谁没有**:FT QC 永远没有 (因为 oracle 必须先被测进 quantum register)
- **应用**:物理样本检测 / 物理 trap probing / 现实世界的反事实感知
- **规模影响**:跟 FT QC 不竞争 · 形成 co-processor 关系

### 超越点 2 · D1 子维度 · adversary-undetectable 查询

- **谁有**:$\Phi^{CF}_f$ 在 $\delta \to 0$ 极限
- **谁没有**:FT QC 即使无限大也没有 (unitary oracle calls 必然纠缠 → 可测 → 可被检测)
- **应用**:对抗性 reconnaissance / 不可察觉 audit / 涉密系统的 stealth probe
- **规模影响**:FT QC 量产后我们仍然独占

### 超越点 3 · D2 · 硬件成本-能力曲线

- **谁更便宜**:我们 ~6 个数量级
- **延续性**:FT QC 量产后差距会缩小但不会消失 (类比模拟光学 FFT vs 数字 FFT)
- **应用**:任何需要 counterfactual 但量级 N 适中 (≤100) 的场景
- **规模影响**:FT QC 普及前的 10-20 年时间窗口 · 我们占 niche · 之后退到 hybrid co-processor 模式

## 6 · 时间窗口分析

模式回顾:专用先到 · 占 niche · 通用后追 · 部分被吃但 narrative 留下。

我们处在 counterfactual computing 的最早期。对手不是其他 photonic IFM 团队 (全球也就 3-5 组活跃) · 是 **FT QC 的成熟速度**。

### 各阶段的相对地位

| 阶段 | FT QC 状态 | 我们的地位 |
|---|---|---|
| 现在 | 没成熟 | 专用方案是唯一可行实现 · 占 niche 黄金期 |
| FT QC 早期可用 (50-100 logical qubits) | 仅能做 demo · 不能解 100+ N 问题 | 仍然唯一可行 |
| FT QC 中期 (1k logical qubits) | 能模拟我们 · 但成本仍极高 | 我们在 D2 上仍然便宜 · D3 上仍然独占 |
| FT QC 后期 (普及) | 可以替代我们做 D1 工作 | 我们退守 D3 (外部物理 oracle) · 永久占有 |

整条时间线我们都不会被完全淘汰。**永久独占** D3。

## 7 · 战略含义

不要号称 "我们的算子超越 FT QC" · 容易被学术界笑话。

应该号称:

- "我们用专用 photonic 硬件 · 在 FT QC 普及前 · 提前解锁了 counterfactual function evaluation 算子"
- "我们独占 FT QC 永远到不了的 interface domain (外部物理 oracle)"
- "我们提供 FT QC 即便无限大也做不到的 adversary-undetectable 查询能力"

这三句都站得住 · 都不夸张 · 都有 narrative。
