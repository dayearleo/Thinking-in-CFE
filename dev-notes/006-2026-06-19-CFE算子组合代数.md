> **状态**:算子组合 v1 · 5 种基础组合定义完整 · cost composition theorem 仍为草稿 · 算子代数公理化为开放问题

## 0 · 起源

`dev-notes/004` 定义单个算子 $\Phi^{CF}_f$ · `dev-notes/005` 论证其超越点 —— 但单个算子的真正威力是 **可组合**。本文件定义 5 种基础组合操作 + 推导 cost composition + 列出未完成的开放问题。

## 1 · 为什么需要组合代数

单算子 $\Phi^{CF}_f$ 只能算单个 Boolean 函数 · 现实问题通常需要:

- 多步推理 (分类 → 排序 → 验证)
- 条件分支 (if hit then deep test else skip)
- 嵌套查询 (反事实判断 "如果 X 那么 Y 的反事实结果是什么")
- 迭代收敛 (Grover-like 迭代)

如果没有组合规则 · 算子只是一次性原语 · 工程价值有限。组合代数让算子成为**可编程构件**。

## 2 · 5 种基础组合

设 $\Phi_f, \Phi_g, \Phi_h$ 是反事实算子 · 各自作用于 oracle 集合 $\mathcal{O}_f, \mathcal{O}_g, \mathcal{O}_h$。

### C1 · 串行 (Sequential)

$$\Phi_g \circ \Phi_f$$

**操作**:
- 先跑 $\Phi_f(\mathcal{O}_f) \to y_f$
- 把 $y_f$ 合成为一个新 oracle $\tilde{O}_{y_f}$ 接入 $\Phi_g$
- 跑 $\Phi_g(\mathcal{O}_g \cup \{\tilde{O}_{y_f}\}) \to y$

**语义**:前一个算子的结果作为后一个的输入 bit · 编进 oracle 配置。

**用例**:multi-stage 过滤 (先粗筛再精筛)

### C2 · 并行 (Parallel)

$$\Phi_f \parallel \Phi_g$$

**操作**:
- 同时跑 $\Phi_f(\mathcal{O}_f) \to y_f$ 和 $\Phi_g(\mathcal{O}_g) \to y_g$ (oracle 集合不相交)
- 经典后处理 $h(y_f, y_g) \to y$

**语义**:不同 oracle 子集独立运行 · 不共享量子 state · 结果经典合并。

**用例**:多 region 独立检测后汇总

### C3 · 嵌套 (Nested · 最强)

$$\Phi_f[\Phi_g]$$

**操作**:
- $\Phi_f$ 的某个 oracle $O_i$ 由 $\Phi_g$ 的"虚拟 oracle"提供
- 虚拟 oracle 的"obstacle 在不在"由 $\Phi_g$ 的反事实结果决定
- 全程不真测 $\Phi_g$ 的实际 oracle

**语义**:**反事实地评估 "假如 $\Phi_g$ 结果是 $X$ · 那么 $\Phi_f$ 的值是什么"** —— 全程不真触发 $\Phi_g$ 的 oracle 也不真触发 $\Phi_f$ 的 oracle。

**用例**:counterfactual lookahead · 反事实反事实推理

**警告**:嵌套深度大时 $\delta$ 累积可能失控 (见 §4)。

### C4 · 条件 (Conditional)

$$\Phi_g \;|\; \Phi_f \;:\; \Phi_h$$

**操作**:
- 跑 $\Phi_f(\mathcal{O}_f) \to y_f$
- if $y_f = 1$: 跑 $\Phi_g$ → 返回其结果
- else: 跑 $\Phi_h$ → 返回其结果

**语义**:分支算子树。注意未走的分支 oracle 不被触发 (省一支代价)。

**用例**:if-then-else 决策 · adaptive algorithm 主干

### C5 · 迭代 (Iterative)

$$\Phi_f^k$$

**操作**:
- $\text{state}_0 = |\psi_0\rangle$
- for $j = 1..k$: $\text{state}_j = \Phi_f(\text{state}_{j-1}, \mathcal{O})$
- 输出 $\text{state}_k$ 的测量结果

**语义**:类比 Grover 迭代 · 但每次迭代反事实 —— oracle 不被消耗 (在合适 $\delta$ 下)。

**用例**:amplitude amplification 的反事实版本 / 反事实搜索定位

## 3 · Cost Composition Theorems (草稿)

记 $\delta_{\text{tot}}$ 为整个组合算子对原始 oracle 的**总触发概率上界** · $B_{\text{tot}}$ 为总 oracle 调用次数上界。

### 串行 (C1)

$$\delta_{\text{tot}}(\Phi_g \circ \Phi_f) \leq \delta_f + \delta_g$$
$$B_{\text{tot}}(\Phi_g \circ \Phi_f) = B_{\delta_f}(f) + B_{\delta_g}(g)$$

(加性 · 失败率叠加 · 调用次数相加)

### 并行 (C2)

$$\delta_{\text{tot}}(\Phi_f \parallel \Phi_g) = \max(\delta_f, \delta_g)$$
$$B_{\text{tot}}(\Phi_f \parallel \Phi_g) = B_{\delta_f}(f) + B_{\delta_g}(g)$$

(取最大 · 独立事件 · 调用相加但物理并行)

### 嵌套 (C3)

$$\delta_{\text{tot}}(\Phi_f[\Phi_g]) \leq \delta_f \cdot N_{\text{deep}} + \delta_g$$
$$B_{\text{tot}}(\Phi_f[\Phi_g]) = B_{\delta_f}(f) \cdot B_{\delta_g}(g)$$

(深度放大 · 危险 · $N_{\text{deep}}$ 是嵌套深度;调用次数相乘 · 指数膨胀)

### 条件 (C4)

$$\delta_{\text{tot}} \leq \delta_f + \max(\delta_g, \delta_h)$$
$$B_{\text{tot}} = B_{\delta_f}(f) + \max(B_{\delta_g}(g), B_{\delta_h}(h))$$

(条件分支只走一支)

### 迭代 (C5)

$$\delta_{\text{tot}}(\Phi_f^k) \leq k \cdot \delta_f$$
$$B_{\text{tot}}(\Phi_f^k) = k \cdot B_{\delta_f}(f)$$

(线性累积)

### 警告

上述全部为**合理估计** · 不是严格证明。严格证明需要建立在 quantum query model + Zeno scheme 上的形式分析。**这是开放工作**。

## 4 · 嵌套深度引起的 $\delta$ cascade · 工程上要怎么控

C3 嵌套时 $\delta_{\text{tot}} \propto N_{\text{deep}}$ · 深度 5 就让 $\delta = 10^{-3}$ 变成 $5 \times 10^{-3}$ · 深度 100 就变成 $10^{-1}$ · 反事实性丢光。

控制策略 (类比经典 cascading failure mitigation):

- **Reset 屏障**:每嵌套 $k$ 层强制 fresh oracle reload · 重置 $\delta$ 累积
- **预算分配**:把总 $\delta_{\text{budget}}$ 按嵌套深度分配 · 每层 $\delta_i = \delta_{\text{budget}}/(N_{\text{deep}} \cdot \text{depth weight})$
- **Adaptive 阈值**:嵌套到一定深度后切换到 standard quantum query (放弃反事实保证 · 保住正确性)
- **拒采样**:如果检测到 $\delta$ 超预算 · 整个 chain 作废重跑 · 跟分布式系统的 retry 类似

这些都是新工程问题 · 没有现成 best practices · 是真正可贡献的工程空间。

## 5 · Programming Model / IR · 草案

为让算子代数能被算法学家直接用 · 需要 programming model 抽象。

### 5.1 · 表达式层

```
expr  := Φ(f, [oracle_1, ..., oracle_N], δ, ε)
      | expr ∘ expr             // sequential
      | expr ‖ expr             // parallel
      | expr [oracle ← expr]    // nested
      | expr ? expr : expr      // conditional
      | iter(expr, k)           // iterative
      | const_oracle(value)     // 经典 bit 包装成 oracle
```

### 5.2 · 类型系统层

- Type `Oracle[T]` · 类型化 oracle (T 是被探测属性)
- Type `Probe[N, δ]` · N 路 probe · 触发率上界 $\delta$
- Type `Result[ε]` · 结果 · 错误率 $\epsilon$
- 编译期检查 $\delta$/$\epsilon$ 是否在预算内 (类似 Rust borrow checker)

### 5.3 · 编译器优化层

可以做的优化:

- **代数简化**:$\Phi_{OR}(\Phi_{OR}(a, b), c) = \Phi_{OR}(a, b, c)$
- **嵌套展开**:把 $\Phi_f[\Phi_g]$ 在深度小时展开为 $\Phi_{f \circ g}$
- **预算分配**:自动把 $\delta_{\text{budget}}$ 分配给每个子算子
- **死代码消除**:条件分支检测到不会走到的子树直接消除

### 5.4 · 后端目标

- **Photonic chip backend** · 编译到具体的集成芯片 (mode 数 / EAM 配置)
- **FT QC backend** · 当 FT QC 可用 · 编译到 quantum circuit (供未来用)
- **Simulator backend** · classical simulator 用于算法 prototyping

## 6 · 已实现库 vs 真空地带 (现状)

**已存在 (学术)**:

- Bomb query complexity 形式化 (Lin & Lin 2015) · 只覆盖单算子
- IFM 单链路实物 / 多对象 IFM 实物 (Hance 2025)

**真空地带 (没人做)**:

- 工程级嵌套深度控制策略

## 7 · 开放问题清单

- **Q1 · 严格 cost composition**:证明 §3 的上界 (或给出 tight bound)
- **Q2 · 嵌套深度上限**:在给定 $\delta_{\text{budget}}$ 下最多能嵌套多深?是否存在 lower bound 表明深度有本质极限?
- **Q3 · 算子代数公理化**:能否给 $\Phi^{CF}$ 一组公理 (类似线性代数 / 关系代数) · 使所有 cost 定理可推导?
- **Q4 · 等价性判定**:两个组合表达式 $E_1, E_2$ 等价 (输出分布相同 + 触发率相同) 是否可判定?
- **Q5 · 最优编译**:给定一个算法 expression · 找成本最小的等价表达式 (compiler optimization)
- **Q6 · 跟标准量子电路的 interop**:能否在 quantum circuit 中嵌入 $\Phi^{CF}$ 作为 sub-routine? 如果可以 · 跟 standard quantum query 怎么 hybrid?
- **Q7 · 类型系统设计**:Linear types / affine types / 量子专用 type system 哪个更合适?

每个问题都可以是一篇论文。
